# Secrets Setup Guide

**v1.0.0** — Required secrets for production deployment.

---

## Overview

This guide lists every secret, API key, and sensitive configuration required to run the platform in production. Secrets are never hardcoded. They are injected via environment variables at deploy time.

---

## 1. JWT_SECRET_KEY

| Property | Value |
|----------|-------|
| **Required** | ✅ Yes |
| **Platform** | Backend (all deployments) |
| **Config Location** | `backend/.env` → `JWT_SECRET_KEY` |
| **Default** | `change-me-to-a-random-secret-key` |
| **Category** | ⚫ Secret |

**Generate:**
```bash
# 32-byte hex key (recommended)
openssl rand -hex 32

# Or use Python
python -c "import secrets; print(secrets.token_hex(32))"
```

**Configure:**
```bash
# Local .env
JWT_SECRET_KEY=<generated-64-char-hex>

# Render → Environment Variables → JWT_SECRET_KEY (sync: false — manual entry)

# Docker Compose
JWT_SECRET_KEY=${JWT_SECRET_KEY}  # passed from host env
```

**Failure if missing:** Backend starts but logs a warning. All JWT tokens are signed with the insecure default — anyone can forge tokens.

## 2. GEMINI_API_KEY

| Property | Value |
|----------|-------|
| **Required** | ✅ Yes (if AI_PROVIDER=gemini) |
| **Platform** | Backend |
| **Config Location** | `backend/.env` → `GEMINI_API_KEY` |
| **Default** | (empty) |
| **Category** | ⚫ Secret |

**Obtain:**
1. Visit https://aistudio.google.com/apikey
2. Sign in with Google account
3. Create API key (Gemini 2.0 Flash — free tier available)

**Configure:**
```bash
# Local .env
GEMINI_API_KEY=AIzaSy...

# Render → Environment → GEMINI_API_KEY (sync: false)

# Docker Compose
GEMINI_API_KEY=${GEMINI_API_KEY}
```

**Failure if missing:** AI provider health check fails (`/ready` shows `ai_provider: fail`). Chat and RAG features return errors.

## 3. DATABASE_URL

| Property | Value |
|----------|-------|
| **Required** | ✅ Yes |
| **Platform** | Backend |
| **Config Location** | `backend/.env` → `DATABASE_URL` |
| **Default** | `postgresql://healthcare_user:healthcare_pass@localhost:5432/healthcare_agent` |
| **Category** | ⚫ Secret |

**Format:**
```
postgresql://<user>:<password>@<host>:<port>/<dbname>
```

**Configure per platform:**

| Platform | Example |
|----------|---------|
| Local Docker | `postgresql://healthcare_user:healthcare_pass@localhost:5432/healthcare_agent` |
| Docker Compose | `postgresql://healthcare_user:healthcare_pass@postgres:5432/healthcare_agent` |
| Render | Auto-injected via `fromDatabase` reference in `render.yaml` |
| Neon (free) | `postgresql://user:pass@ep-xxx.us-east-2.aws.neon.tech/healthcare_agent?sslmode=require` |

**Failure if missing:** Application fails to start — database connection rejected.

## 4. POSTGRES_USER / POSTGRES_PASSWORD / POSTGRES_DB

| Property | Value |
|----------|-------|
| **Required** | ✅ Yes (Docker Compose) |
| **Platform** | Local Docker / Docker Compose |
| **Config Location** | `docker-compose.yml` / `docker-compose.production.yml` |
| **Default** | `healthcare_user` / `healthcare_pass` / `healthcare_agent` |
| **Category** | 🔴 Required / ⚫ Secret |

**Change defaults for production:**
```bash
export POSTGRES_USER=prod_healthcare
export POSTGRES_PASSWORD=<strong-random-password>
export POSTGRES_DB=healthcare_agent
docker compose -f docker/docker-compose.production.yml up -d
```

**Render:** Database credentials are auto-generated by the Render PostgreSQL service. View in Render Dashboard → Database → healthcare-db → Connections.

## 5. BACKEND_CORS_ORIGINS

| Property | Value |
|----------|-------|
| **Required** | ✅ Yes |
| **Platform** | Backend |
| **Config Location** | `backend/.env` → `BACKEND_CORS_ORIGINS` |
| **Default** | `http://localhost:3000,http://localhost:5173` |
| **Category** | 🔴 Required |

**Set for production:**
```bash
# Single domain
BACKEND_CORS_ORIGINS=https://healthcare-frontend.onrender.com

# Multiple domains (comma-separated)
BACKEND_CORS_ORIGINS=https://healthcare-frontend.onrender.com,https://your-custom-domain.com
```

**Render:** `BACKEND_CORS_ORIGINS: https://healthcare-frontend.onrender.com` (in `render.yaml`)

## 6. CHROMA_HOST / CHROMA_PORT

| Property | Value |
|----------|-------|
| **Required** | ✅ Yes |
| **Platform** | Backend |
| **Config Location** | `backend/.env` → `CHROMA_HOST`, `CHROMA_PORT` |
| **Default** | `localhost:8001` |
| **Category** | 🔴 Required |

**Configure per platform:**

| Platform | CHROMA_HOST | CHROMA_PORT |
|----------|-------------|-------------|
| Docker Compose | `chromadb` (service name) | `8000` |
| Render (same service) | `localhost` | `8000` |
| External VPS | `<vps-ip>` | `8000` |

**Note:** `.env.example` defaults to port `8001` (to avoid conflict with Render backend on `8000`). In `docker-compose.production.yml` and `render.yaml`, port `8000` is used. Ensure consistency.

## 14. Frontend Secrets

| Variable | Required | Location | Notes |
|----------|----------|----------|-------|
| `NEXT_PUBLIC_API_URL` | ✅ Yes | `frontend/.env.local` | Public — visible in browser |
| `NEXT_PUBLIC_WS_URL` | 🟡 | `frontend/.env.local` | WebSocket endpoint |
| `NEXT_PUBLIC_APP_URL` | ✅ Yes | `frontend/.env.local` | Self URL for redirects |

Frontend env vars with `NEXT_PUBLIC_*` prefix are bundled into the JavaScript at build time. They are visible to end users — never put real secrets here.

## Quick Start — First Deployment

```bash
# 1. Generate JWT secret
JWT_SECRET=$(openssl rand -hex 32)

# 2. Set on Render
render env set healthcare-backend JWT_SECRET_KEY "$JWT_SECRET"
render env set healthcare-backend GEMINI_API_KEY "AIzaSy..."

# 3. Deploy
git push origin main
```
